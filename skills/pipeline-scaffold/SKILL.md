# Pipeline Scaffold

Create the standard project structure. Called by `/pipeline-new`.

**Input:** `$ARGUMENTS` = project name or description

GSD installed: !`which gsd 2>/dev/null && echo "YES" || echo "NO"`

## Steps

### 1. Initialize git

```bash
git init 2>/dev/null || true
```

If git user.name or user.email aren't configured, warn the user but continue.

### 2. Create directory tree

```bash
mkdir -p docs .gsd/designs .claude
```

### 3. Initialize GSD (if installed)

If GSD is installed, run:
```bash
gsd headless "init" --no-session
```

GSD creates `.gsd/CODEBASE.md` (project file map) and `.gsd/STATE.md`.

### 4. Create pipeline-owned files

These files are managed by the pipeline, not GSD. Always create them:

- `.gsd/KNOWLEDGE.md` — `# Knowledge` heading only (pipeline populates during init/build)
- `.gsd/REQUIREMENTS.md` — `# Requirements` heading only (populated during office hours)

### 5. Write CLAUDE.md

Use the Write tool. Structure:

```markdown
# [Project Name]

[One-line description from $ARGUMENTS]

## Tech Stack

TBD — fill in after initial setup.

## Quick Start

TBD — fill in after first build.

## Pipeline Commands

| Command | When | What happens |
|---------|------|--------------|
| `/pipeline-build` | New feature | Requirements → brainstorm → spec → plan → design → execute → QA → docs |
| `/pipeline-quick` | Small fix | Superpowers handles it, conditional doc update |
| `/pipeline-qa` | Run QA | Code review + security + browser testing + unit tests |

## Skill Routing

When these patterns appear, invoke the corresponding skill:

- "brainstorm", "design", "plan a feature" → `Skill(superpowers:brainstorming)`
- "review this PR", "code review" → `Skill(gstack:review)`
- "security audit", "check for vulnerabilities" → `Skill(gstack:cso)`
- "test this site", "QA", "find bugs" → `Skill(gstack:qa)`
- "debug this", "why is this broken" → `Skill(gstack:investigate)`
- "ship", "create PR", "deploy" → `Skill(gstack:ship)`
- "be careful", "production work" → `Skill(gstack:careful)`

## Documentation

- `docs/ARCHITECTURE.md` — system design, data flow, key directories
- `docs/API.md` — endpoints, params, responses
- `docs/COMPONENTS.md` — UI component inventory
- `docs/MODELS.md` — data shapes, relationships
- `.gsd/TESTING.md` — test strategy (generated by pipeline)

## Testing

See `.gsd/TESTING.md` for test strategy. Run `/pipeline-qa` for full QA suite.

## Conventions

- Commits: conventional commits (feat:, fix:, docs:, chore:)
- Testing: TDD enforced by Superpowers
- Other conventions: TBD — fill in as patterns emerge
```

Keep under 80 lines. This is an index.

### 6. Write doc stubs

Use the Write tool for each:

- `docs/ARCHITECTURE.md` — `# Architecture\n\nSystem design, data flow, and where things live.`
- `docs/API.md` — `# API\n\nEndpoints, parameters, and responses.`
- `docs/COMPONENTS.md` — `# Components\n\nUI component inventory with file paths.`
- `docs/MODELS.md` — `# Models\n\nData shapes, types, and relationships.`

### 7. Write VERSION file

```
0.0.0
```

Used by gstack `/ship` for version bumping.

### 8. Update .gitignore

Append if not present:

```
.claude/pipeline-state.json
node_modules/
.DS_Store
```

### 9. Initial commit

```bash
git add -A
git commit -m "chore: scaffold project with pipeline infrastructure"
```

Capture the commit SHA — the caller needs it as a base ref for doc updates later.

## Done

Report what was created and the commit SHA. Do NOT proceed to the next phase — the caller handles routing.
